**MVC (Model-View-Controller)** — это схема разделения ответственности в приложении. Представь, что ты пришел в ресторан:
1. **Model (Модель)** 🥩 — это кухня. Здесь хранятся данные и правила их обработки (бизнес-логика). Модель не знает, как выглядит меню или кто заказывает еду, она просто «готовит» данные.
2. **View (Представление)** 📋 — это меню и сервировка. То, что видит клиент. В случае с API «представлением» обычно является **JSON**-ответ.
3. **Controller (Контроллер)** 👨‍🍳 — это официант. Он принимает твой заказ (HTTP-запрос), передает его на кухню (Модель) и приносит тебе готовое блюдо (View).

### Как это выглядит в Spring Boot? 🍃
Spring Boot делает создание таких систем очень быстрым. Для API мы используем специальный тип контроллера — `@RestController`. Он автоматически превращает объекты, которые возвращает Java, в JSON-формат для клиента.
Чтобы создать простой API, нам обычно нужны три вещи:
- **Класс-данных** (POJO/Record) — наша Модель.
- **Контроллер** — класс с аннотациями для обработки путей (например, `/hello`).
- **Gradle** — инструмент, который соберет все зависимости (библиотеки) воедино.

**REST API** (Representational State Transfer) — это архитектурный стиль взаимодействия между программами. Если совсем просто: это набор правил о том, как сервер и клиент (например, твое приложение на Android и бэкенд на Spring) должны обмениваться данными.
### Главные принципы REST
Чтобы API считалось «RESTful», оно должно следовать нескольким правилам:
#### 1. Единообразие интерфейса

Все ресурсы (данные) имеют свои адреса (URI). Мы обращаемся не к «функциям», а к «сущностям».
- **Плохо:** `http://site.com/getAllTasks` (это стиль RPC)
- **Хорошо (REST):** `http://site.com/tasks`
#### 2. Клиент-Сервер (Client-Server)
Клиент (фронтенд) и сервер (бэкенд) независимы. Ты можешь полностью переписать мобильное приложение на Kotlin, но серверу на Spring будет все равно, пока формат запросов остается прежним.
#### 3. Отсутствие состояния (Stateless)
Сервер не должен помнить, что ты делал секунду назад. Каждый запрос должен содержать **всю** необходимую информацию (например, токен авторизации). Сервер не хранит «сессию» в памяти.

#### Коды ответов (Status Codes)
Сервер всегда отвечает числовым кодом, чтобы клиент сразу понял результат:
- **200 OK** — все прошло успешно.
- **201 Created** — ресурс (задача) успешно создан.
- **400 Bad Request** — ты прислал неправильные данные (например, пустое название).
- **401 Unauthorized** — ты не залогинен.
- **404 Not Found** — такой задачи не существует.  
- **500 Internal Server Error** — ошибка в коде на сервере.
### Формат данных: JSON 📄
В 99% случаев современные REST API используют формат **JSON**. Это текстовый формат, который легко читается и человеком, и машиной.
```json
{ "id": 1,
 "title": "Выучить базу REST", 
 "completed": false 
 }
```
